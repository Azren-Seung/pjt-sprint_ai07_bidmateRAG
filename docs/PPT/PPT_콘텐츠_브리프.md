### 슬라이드 5 — 파이프라인 & 정직한 KPI 5축 (Visual Spine 원본)

**핵심 텍스트**:
- 전체 파이프라인: 파싱 → 청킹 → 임베딩 → 검색 → 생성 → 평가
- KPI 5축으로 Retrieval·Generation·Operational 영역을 나누어 측정
- 이 Spine은 뒤에서 각 파트 시작 슬라이드(8/13/18/24)에서 현재 위치를 하이라이트하며 재등장

**수치/표**:
| KPI | 목표 | 실측 (80문항) | 판정 |
|---|---|---|---|
| MRR | ≥ 0.70 | 1.00 | ✅ 초과달성 |
| Faithfulness | ≥ 0.85 | 0.81 | ⚠️ 근접 |
| Answer Correctness | ≥ 0.75 | (실측 반영) | ⚠️ 확인 |
| Latency | ≤ 10초 | 20.7초 | ❌ 미달 |
| Cost/건 | ≤ $0.05 | $0.008 | ✅ 달성 |

**시각화 제안**: 6단 가로 파이프라인 다이어그램 + 하단 KPI 5축 표

**각주**: bench-3905e4ce, 2026-04-19, 80문항 full. "Retrieval은 천장 도달, Generation·Latency는 개선 여지"

### 슬라이드 6 — "MRR 0.000 → 1.000" Judge v2 드라마 & 구조 ★

> **Hook**: "측정 도구의 신뢰성이 곧 성능 지표의 신뢰성"

**핵심 텍스트**:
- Judge v1 시절: LLM이 Pass/Fail 주관 응답 → 4회 연속 MRR=0.000 기록 (채점 파싱 실패)
- Judge v2 전환 후 즉시 MRR 1.000 정상화 — 파이프라인이 아니라 측정 도구가 문제였음
- v2 구조: LLM은 근거(Evidence)·추론(Reasoning)만 추출 / 점수는 코드가 5단계 비율제(0/0.25/0.5/0.75/1.0) 계산

**수치/표**:
| 항목 | v1 | v2 |
|---|---|---|
| 채점 방식 | LLM 주관 점수 | 코드 비율제 계산 |
| 지표 | Pass/Fail | 5대 ratio score |
| 결과 | 4회 연속 MRR 0.000 | 즉시 MRR 1.000 |
| 투명성 | 불가 | 채점 과정 로깅 |

**시각화 제안**: 시계열 꺾은선 그래프 (날짜 × MRR) + v1/v2 구조 대비 다이어그램

**각주**: 커밋 4867df2 `feat: add LLM judge v2 with evidence-based ratio scores`

### 슬라이드 7 — "평가셋 오답률 30% → 10%" 데이터 무결성 혁신 ★

> **Hook**: "좋은 Judge가 있어도 평가셋이 엉망이면 아무 의미 없다"

**핵심 텍스트**:
- 팩트체크 기반 문제 생성: `ground_truth_docs` + `source_pages` + `reasoning_process` 3컬럼 강제
- 생성 AI × 검수 AI 교차 검증 파이프라인으로 평가셋 자체의 무결성 확보
- 정답 존재 여부를 사전 검증해야 Judge가 의미 있는 신호 측정 가능
- 최종 평가셋: Q001~Q080 (eval_batch_35.csv, Type A~E 분류)

**수치/표**:
| 항목 | Before | After |
|---|---|---|
| 평가셋 오답률 | 30% | **10%** (3배 개선) |
| 문항 수 | 80 | 80 (동일) |
| 검수 방식 | 단일 생성 | 생성 AI × 검수 AI 교차 |

**시각화 제안**: Before/After 막대 그래프 (오답률) + 팩트체크 3컬럼 강제 스크린샷

**각주**: 데이터 베이스라인 data/eval/eval_v1/eval_batch_35.csv, Type A~E 포함
